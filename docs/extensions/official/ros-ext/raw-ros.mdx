# Raw ROS

Raw ROS mode lets ReductROS decode ROS messages that are stored as individual records rather
than as MCAP or rosbag archives. It is intended for pipelines such as `reduct-bridge`, which write
message payloads with content type `application/ros1` and store the schema metadata required for
decoding in the `$ros` attachment.

## Supported input

The Raw ROS extractor expects:

- records stored with content type `application/ros1`
- one ROS 1 message per record
- one ROS topic per entry
- a `$ros` attachment on the entry that provides the schema metadata

Unlike the MCAP and rosbag modes, Raw ROS supports **extraction only**. It does not support
transforming records into new bag or archive files.

## Required $ros attachment

The extractor requires an attachment named `$ros`. The attachment must contain the following JSON
fields:

| Field         | Type     | Description                                                      |
| ------------- | -------- | ---------------------------------------------------------------- |
| `encoding`    | `string` | Must be `ros1`.                                                  |
| `schema`      | `string` | Full ROS 1 message definition used to decode the binary payload. |
| `topic`       | `string` | Topic name associated with the stored records.                   |
| `schema_name` | `string` | ROS 1 message type, for example `std_msgs/String`.               |

If the records were ingested with `reduct-bridge`, this attachment is written automatically.

**Example `$ros` metadata**

```json
{
  "encoding": "ros1",
  "schema": "string data",
  "topic": "/chatter",
  "schema_name": "std_msgs/String"
}
```

## Query format

```json
{
  "ext": {
    "ros": {
      "extract": {
        "encode": {
          "data": "base64"
        },
        "as_label": {
          "frame_id": "header.frame_id"
        }
      }
    }
  }
}
```

:::info
The query config contains only extraction options. The ROS schema and topic information come from
the `$ros` attachment.
:::

## Data extraction

The `extract` property tells ReductROS how to decode each raw ROS 1 record and how to shape the
resulting JSON output.

| Parameter        | Type     | Mandatory | Description                                                                                      |
| ---------------- | -------- | --------- | ------------------------------------------------------------------------------------------------ |
| `encode`         | `object` | No        | A map of binary fields to conversion formats.                                                    |
| `encode.<field>` | `string` | No        | Conversion format for the field. Supported values are `base64` and `jpeg`.                       |
| `as_label`       | `object` | No        | Computed labels extracted from the decoded JSON payload. Use JSON paths without the `$.` prefix. |

Raw ROS entries do not need a `topic` parameter in `extract`. Each entry represents a single topic,
and the topic name comes from the `$ros` attachment.

### Encoding binary fields

The `encode` property controls how byte arrays are represented in the JSON output.

| Format   | Description                                         |
| -------- | --------------------------------------------------- |
| `base64` | Encodes the selected byte array as a base64 string. |
| `jpeg`   | Encodes image bytes as a base64-encoded JPEG.       |

:::info
Currently, JPEG conversion is supported only for `sensor_msgs/Image` messages with `encoding` set
to `rgb8`, `bgr8`, or `mono8`.
:::

### Timestamps

The extractor uses `header.stamp.sec` and `header.stamp.nsec` when they are present in the decoded
message. If the message has no ROS header, the output record keeps the original ReductStore record
timestamp.

## Output behavior

- One output record is produced per input raw ROS record.
- The output entry name is preserved.
- Output payload is `application/json`.
- Original input labels are preserved.
- Existing computed labels are preserved.
- The extractor adds the following computed labels:
  - `encoding`
  - `schema`
  - `topic`
  - labels defined via `as_label`

## Examples

The following examples use Raw ROS sample data downloaded from the local `ros-it-2` bucket.
Although the examples are written in Python, the same query structure works with any official SDK.

### Extracting messages as JSON

This example writes raw `geometry_msgs/Point` records together with the `$ros` attachment and then
extracts them as JSON. It also creates a computed label from the decoded `x` field.

#### Expected output

```text
Record timestamp: 1773324549469334
Record labels: {'@coord_x': '2.0', '@encoding': 'ros1', '@schema': 'geometry_msgs/Point', '@topic': '/sensor/pos', 'x': '2.0'}

```

#### Explanation

- The example uploads three raw ROS 1 records to a new entry.
- It writes the `$ros` attachment before querying the entry.
- The extractor decodes the binary payload into a JSON `geometry_msgs/Point` message.
- The `as_label` option exposes the decoded `x` field as `@coord_x`.
- The `when` filter returns only the record where `@coord_x` equals `2.0`.
